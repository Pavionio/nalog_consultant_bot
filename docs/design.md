# Design-Doc: Чат-бот — налоговый консультант

Документ описывает архитектуру, данные, модель и эксплуатацию RAG-чат-бота для налоговых консультаций МСП. Краткое описание проекта и инструкция по запуску — в [README](../README.md). Экспериментальное обоснование выбора компонентов — в отчёте `text/main.tex`.

## 1. Контекст проекта
### 1.1 Бизнес-задача
Малый и средний бизнес в Российской Федерации регулярно сталкивается с необходимостью получения налоговых консультаций: выбор системы налогообложения, расчёт налогов, отчётность, льготы, изменения в законодательстве. При этом:
* услуги профессиональных налоговых консультантов стоят дорого;
* информация в открытых источниках часто разрозненная, устаревшая или противоречивая;
* предприниматели тратят значительное время на поиск и интерпретацию информации.

Цель проекта — создание интеллектуального чат-бота, который позволяет в диалоговом режиме получать обоснованные и актуальные ответы на вопросы по налоговой тематике, основанные на нормативных документах и справочных материалах.
Использование RAG-подхода оправдано тем, что:
* налоговая информация должна быть привязана к источникам;
* необходимо уменьшить риск галлюцинаций языковой модели;
* требуется возможность оперативного обновления базы знаний без переобучения модели.

Критерий успеха проекта — способность системы:
* корректно отвечать на типовые налоговые вопросы,
* ссылаться на релевантные источники,
* снижать время поиска информации пользователем.

### 1.2 Целевая аудитория и пользователи
Целевая аудитория:
* индивидуальные предприниматели,
* владельцы малого и среднего бизнеса,
* начинающие бухгалтеры и предприниматели.

Сценарии использования:
* текстовый диалог в режиме «вопрос — ответ» в Telegram,
* получение справок, пояснений и выдержек из нормативных документов.

Предполагаемая нагрузка:
* десятки-сотни сессий в день,
* пиковая нагрузка — в периоды отчётности.

### 1.3 Ограничения и допущения
Ограничения:
* чат-бот не даёт персонализированные юридически значимые рекомендации, а даёт справочную информацию (с явным уведомлением об этом при старте и в правилах);
* система не выполняет действий от имени пользователя (подача отчётности, расчёт налогов, отправка деклараций);
* **бот отвечает на каждый вопрос за один шаг** (один retrieval + одна генерация) и не ведёт многошаговый диалог-расследование: он не декомпозирует сложный вопрос на подзадачи и не задаёт цепочку уточняющих вопросов (см. §5.2).

Допущения:
* база знаний состоит только из открытых и официальных источников;
* бот отвечает только в рамках налоговой тематики и отказывается от оффтоп-вопросов;
* персональные данные пользователей не собираются и не обрабатываются.

## 2. Архитектура решения
### 2.1 Общая схема
Компоненты системы:
* **Telegram-бот** (`src/bot/`, aiogram) — интерфейс взаимодействия с пользователем;
* **RAG-сервис** (`src/bot/rag_service.py` + `src/rag/core.py`) — переформулирует вопрос по истории сессии, выполняет поиск и формирует ответ;
* **RAG-модуль** (`src/rag/`) — эмбеддинг запроса, поиск, реранжирование, сборка контекста, генерация;
* **Векторная БД Qdrant** — хранит эмбеддинги фрагментов документов и payload-метаданные для цитирования;
* **PostgreSQL** — реестр документов и состояние обновления корпуса (`rag_doc`, `rag_fetch_log`);
* **LLM-сервер (llama.cpp)** — генерирует финальный ответ по OpenAI-совместимому API;
* **Пайплайн загрузки** (`fetch/`) — периодический парсинг источников, чанкинг и индексация.

```
Telegram → RagService (session-rewrite) → эмбеддинг (e5-large)
   → Qdrant dense top-50 → reranker (BGE-v2-m3) top-k
   → сборка контекста [1..k] → llama.cpp (Qwen3-14B) → ответ + источники
```

### 2.2 Хранилище знаний / документооборот
Хранение:
* **Qdrant** — векторные представления чанков (коллекция, размерность эмбеддера) и payload (`source_code`, `external_id`, `canonical_url`, `title`, даты, текст чанка) для поиска и цитирования;
* **PostgreSQL** — карточка каждого документа в таблице `rag_doc`: идентификация (`source_code`, `external_id`, `canonical_url`), метаданные (тип, заголовок, даты, номер), отпечатки содержимого (`content_sha256`, `http_etag`, `last_modified`), планирование (`next_check_at`) и связь с Qdrant (`collection`, `doc_key`, `revision`); журнал обращений — `rag_fetch_log`.

Обновление:
* база знаний обновляется периодически (`uv run python -m fetch.init_fetch update`): из `rag_doc` выбираются документы с наступившим `next_check_at`, выполняется повторная загрузка, и при изменении хеша/HTTP-валидаторов документ переиндексируется в Qdrant без полного обхода источников.

### 2.3 Retrieval + Generation
Оптимальный (используемый в боте) пайплайн, выбранный по результатам экспериментов:
* **Предобработка корпуса**: профиль очистки `clean_no_boilerplate` (удаление навигационного шума с защитой юридически значимых строк).
* **Чанкинг**: token-разбиение, размер 1024, перекрытие 256.
* **Эмбеддер**: `intfloat/multilingual-e5-large` (с префиксами `query:` / `passage:`), косинусная близость, нормализованные векторы.
* **Первичный retrieval**: dense-поиск в Qdrant, `fetch_k = 50` кандидатов (гибрид dense+BM25/RRF доступен, но по умолчанию выключен — он помогает не на всех типах запросов).
* **Реранжирование**: cross-encoder `BAAI/bge-reranker-v2-m3` переупорядочивает 50 кандидатов и отдаёт финальный top-k.
* **Сборка контекста**: фрагменты в естественном порядке реранкера (`ctx 1-2-3`), лимит контекста ~18000 символов, нумерация источников `[1]`, `[2]`, …
* **Генерация**: `Qwen/Qwen3-14B` (llama.cpp), режим `no think`, `temperature=0`, системный промпт `strict_citations` (ответ только по контексту, отказ при недостатке данных, структура «вывод — обоснование — источники»).
* **Преобразование запроса**: query rewrite и HyDE по итогам экспериментов не используются (ухудшают retrieval); применяется только переформулировка вопроса по истории диалога (§5.2).

### 2.4 Интеграции и интерфейсы
* Чат-бот в Telegram (aiogram, long-polling).
* LLM — OpenAI-совместимый эндпоинт llama.cpp (`/v1/chat/completions`).
* Qdrant — REST/gRPC; PostgreSQL — через `psycopg`.

### 2.5 Инфраструктура и развёртывание
* Все сервисы поднимаются через `docker compose` (`docker/docker-compose.yml`): PostgreSQL, Qdrant, llama.cpp (CUDA), Telegram-бот.
* Бот и llama.cpp требуют GPU; эмбеддер (e5-large) и реранкер (BGE-v2-m3) загружаются в процессе бота.
* Запуск локально (для разработки) и в облаке с NVIDIA GPU.

## 3. Данные и качество знаний
### 3.1 Сбор и предобработка данных
Источники данных (`config/sources.yaml`, 15 активных источников):
* Налоговый кодекс РФ и нормативные акты (`pravo.gov.ru`),
* письма и разъяснения ФНС России (`nalog.gov.ru`),
* раздел «Ответы на вопросы» Минфина России (`minfin.gov.ru`),
* налоговый календарь ФНС (XML).

Форматы данных: HTML-страницы, XML (календарь), MIME/MHTML.

Шаги предобработки (`fetch/` + `src/rag/text_cleaning.py`):
* извлечение текста из документов (формат-специфичные парсеры);
* очистка текста от служебной информации и навигационного шума (профиль `clean_no_boilerplate`, удаление строк, встречающихся более чем в 5% документов, с защитой юридических терминов);
* разбиение на фрагменты (token, 1024/256);
* нормализация текста (Unicode NFC, схлопывание пробелов);
* дедупликация по `source_code:external_id`;
* добавление метаданных (источник, дата, номер документа, URL).

### 3.2 Векторизация и индексирование
* Эмбеддер `intfloat/multilingual-e5-large` (1024-мерные нормализованные векторы);
* стабильные идентификаторы точек Qdrant через `UUID5(source_code + external_id + chunk_index)`;
* отдельные коллекции под конкретную связку «эмбеддер + чанкинг + профиль очистки» (боевая: `rag_chunks_e5_large_token_1024_256_clean_no_boilerplate`).

### 3.3 Метрики качества знаний
* Retrieval: Hit@k (Hit Rate), MRR@k, nDCG@k, Precision@k, доля hard-negative;
* Генерация: LLM-судья по критериям precision / completeness / format (шкала 1–5) + согласованность с ручной разметкой (Cohen's κ);
* Покрытие, актуальность (даты/редакции в метаданных), консистентность;
* ручная проверка на размеченном «золотом наборе».

## 4. Модель и генерация
### 4.1 Выбор LLM и промптинг
* Используется LLM с поддержкой русского языка — `Qwen/Qwen3-14B` (квантизованный GGUF на llama.cpp), режим `no think`, `temperature=0` (детерминированность и воспроизводимость в legal-tech).
* В промпт (`strict_citations`) передаются:
  * вопрос пользователя;
  * найденный и пронумерованный контекст;
  * инструкция отвечать только по контексту, не выдумывать реквизиты/сроки/номера статей и отказываться при недостатке данных.

### 4.2 Контроль качества ответов
* Если релевантный контекст не найден — бот отказывается отвечать фразой «В предоставленных документах нет достаточной информации для ответа».
* Ответы должны быть:
  * точными и соответствовать найденным источникам;
  * основанными на найденных источниках (с указанием ссылок);
  * нейтральными и профессиональными по тону;
  * краткими и по существу.
* В ответах запрещены: советы по конкретным действиям, категоричные юридические формулировки, выдуманные факты, личные мнения, сторонние темы.
* Ответы и запросы логируются (обезличенно) для последующего анализа (`data/bot_logs_*.jsonl`).

### 4.3 Обучение/дообучение
* Fine-tuning не планируется.
* Улучшение качества достигается улучшением базы знаний, настройкой разбиения и retrieval (см. серии экспериментов в отчёте).

## 5. UX / пользовательский опыт
### 5.1 Сценарии взаимодействия
* Старт диалога: приветствие, правила использования и обязательное принятие правил (`/accept`).
* Вопрос — ответ: пользователь присылает налоговый вопрос обычным сообщением, бот возвращает ответ со ссылками на источники.
* Сброс истории диалога (`/reset`, `/new_sesssion`).
* Отказ в ответе при отсутствии данных или несоответствии тематике бота.

### 5.2 Диалоговая логика
* Поддерживается **multi-turn диалог как контекст**: история сессии (до 20 последних сообщений) используется только для того, чтобы переформулировать текущий вопрос в самостоятельный (session-rewrite) перед поиском.
* **Каждый вопрос обрабатывается за один шаг**: бот не задаёт цепочку уточняющих вопросов, не декомпозирует вопрос на подзадачи и не выполняет многошаговый (multi-hop) поиск. Если контекста недостаточно, бот честно отказывается и может подсказать, как переформулировать вопрос, но не переходит в режим пошагового уточнения.
* Тон общения: нейтральный, профессиональный, вежливый.

### 5.3 Метрики UX
* Доля успешных ответов (не-отказов при наличии релевантного контекста),
* доля корректных отказов,
* среднее время ответа,
* удовлетворённость пользователей (опросы после диалога — 👍 / 👎).

## 6. Безопасность, соответствие и этика
* Персональные данные не запрашиваются и не хранятся; пользователь явно предупреждается не отправлять чувствительную информацию.
* Система не выдаёт юридически значимых рекомендаций (явное уведомление в правилах).
* Пользователь уведомляется, что общается с ботом, и принимает правила использования до начала работы.
* Запросы логируются в обезличенном виде.
* Используются только легальные официальные источники.

## 7. План внедрения и эксплуатации
### 7.1 Этапы проекта
1. Сбор и подготовка данных.
2. Реализация RAG-пайплайна и серия экспериментов по подбору компонентов.
3. Разработка и интеграция Telegram-интерфейса.
4. Тестирование и экспериментальная оценка качества.

### 7.2 Поддержка и эксплуатация
* Обновление базы знаний по мере изменений в законодательстве (планировщик `next_check_at`).
* Мониторинг производительности и качества ответов.
* Логирование ошибок и запросов.
* Регулярные бэкапы томов Qdrant и PostgreSQL.

## 8. Риски и допущения
* **Неполнота базы знаний**: метаданные хранят версию и дату источника; реализовано автоматическое обновление; не охвачены региональные нормативные акты (план — добавить).
* **Сложность юридических формулировок**: приоритет источников по надёжности (НК РФ > письма ФНС/Минфина); строгий промпт против вольных формулировок.
* **Галлюцинации модели**: отказ при отсутствии контекста, `temperature=0`, обязательная привязка к источникам, реранкер для повышения релевантности контекста.
* **Похожие, но нерелевантные документы (hard negatives)**: остаются источником ошибок в налоговом домене (различия по налогу/периоду/категории налогоплательщика) — частично снимаются реранкером.

## 9. Бюджет и ресурсы
* **Человеческие ресурсы**: разработка и сопровождение силами проектной команды; ручная разметка «золотого набора» — разово при калибровке метрик.
* **Технологические ресурсы**: один сервер с NVIDIA GPU (≈16 ГБ VRAM достаточно для Qwen3-14B q4 + e5-large + реранкера); Docker; Qdrant и PostgreSQL — open-source; модели (Qwen3, e5-large, BGE-reranker) — open-source.
* **Примерные затраты**: основная статья — аренда/амортизация GPU-сервера; внешних API-расходов нет (вся генерация и эмбеддинги локальные).
* **ROI**: экономия времени пользователей на поиск и первичную интерпретацию налоговой информации; снижение стоимости первичной консультации относительно платных консультантов.

## 10. Приложения

* Ссылки на специфические документы (см. `config/sources.yaml`).
* Диаграммы архитектуры (см. §2.1).
* Примеры промптов (системный промпт `strict_citations`, промпты rewrite/HyDE — в `src/rag/core.py`).
